      59             : /** FGLocation holds an arbitrary location in the Earth centered Earth fixed
      60             :     reference frame (ECEF). The coordinate frame ECEF has its center in the
      61             :     middle of the earth. The X-axis points from the center of the Earth towards
      62             :     a location with zero latitude and longitude on the Earth surface. The Y-axis
      63             :     points from the center of the Earth towards a location with zero latitude
      64             :     and 90 deg East longitude on the Earth surface. The Z-axis points from the
      65             :     Earth center to the geographic north pole.
      66             : 
      67             :     This class provides access functions to set and get the location as either
      68             :     the simple X, Y and Z values in ft or longitude/latitude and the radial
      69             :     distance of the location from the Earth center.
      70             : 
      71             :     It is common to associate a parent frame with a location. This frame is
      72             :     usually called the local horizontal frame or simply the local frame. It is
      73             :     also called the NED frame (North, East, Down), as well as the Navigation
      74             :     frame. This frame has its X/Y plane parallel to the surface of the Earth
      75             :     (with the assumption of a spherical Earth). The X-axis points towards north,
      76             :     the Y-axis points east and the Z-axis points to the center of the Earth.
      77             : 
      78             :     Since the local frame is determined by the location (and NOT by the
      79             :     orientation of the vehicle IN any frame), this class also provides the
      80             :     rotation matrices required to transform from the Earth centered (ECEF) frame
      81             :     to the local horizontal frame and back. This class "owns" the
      82             :     transformations that go from the ECEF frame to and from the local frame.
      83             :     Again, this is because the ECEF, and local frames do not involve the actual
      84             :     orientation of the vehicle - only the location on the Earth surface. There
      85             :     are conversion functions for conversion of position vectors given in the one
      86             :     frame to positions in the other frame.
      87             : 
      88             :     The Earth centered reference frame is NOT an inertial frame since it rotates
      89             :     with the Earth.
      90             : 
      91             :     The cartesian coordinates (X,Y,Z) in the Earth centered frame are the master
      92             :     values. All other values are computed from these master values and are
      93             :     cached as long as the location is changed by access through a non-const
      94             :     member function. Values are cached to improve performance. It is best
      95             :     practice to work with a natural set of master values. Other parameters that
      96             :     are derived from these master values are calculated only when needed, and IF
      97             :     they are needed and calculated, then they are cached (stored and remembered)
      98             :     so they do not need to be re-calculated until the master values they are
      99             :     derived from are themselves changed (and become stale).
     100             : 
     101             :     Accuracy and round off
     102             : 
     103             :     Given,
     104             : 
     105             :     - that we model a vehicle near the Earth
     106             :     - that the Earth surface radius is about 2*10^7, ft
     107             :     - that we use double values for the representation of the location
     108             : 
     109             :     we have an accuracy of about
     110             : 
     111             :     1e-16*2e7ft/1 = 2e-9 ft
     112             : 
     113             :     left. This should be sufficient for our needs. Note that this is the same
     114             :     relative accuracy we would have when we compute directly with
     115             :     lon/lat/radius. For the radius value this is clear. For the lon/lat pair
     116             :     this is easy to see. Take for example KSFO located at about 37.61 deg north
     117             :     122.35 deg west, which corresponds to 0.65642 rad north and 2.13541 rad
     118             :     west. Both values are of magnitude of about 1. But 1 ft corresponds to about
     119             :     1/(2e7*2*pi) = 7.9577e-09 rad. So the left accuracy with this representation
     120             :     is also about 1*1e-16/7.9577e-09 = 1.2566e-08 which is of the same magnitude
     121             :     as the representation chosen here.
     122             : 
     123             :     The advantage of this representation is that it is a linear space without
     124             :     singularities. The singularities are the north and south pole and most
     125             :     notably the non-steady jump at -pi to pi. It is harder to track this jump
     126             :     correctly especially when we need to work with error norms and derivatives
     127             :     of the equations of motion within the time-stepping code. Also, the rate of
     128             :     change is of the same magnitude for all components in this representation
     129             :     which is an advantage for numerical stability in implicit time-stepping.
     130             : 
     131             :     Note: The latitude is a GEOCENTRIC value. FlightGear converts latitude to a
     132             :     geodetic value and uses that. In order to get best matching relative to a
     133             :     map, geocentric latitude must be converted to geodetic.
     134             : 
     135             :     @see Stevens and Lewis, "Aircraft Control and Simulation", Second edition
     136             :     @see W. C. Durham "Aircraft Dynamics & Control", section 2.2
     137             : 
     138             :     @author Mathias Froehlich
     139             :   */
